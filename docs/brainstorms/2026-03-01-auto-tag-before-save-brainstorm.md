# Auto Tag Before Save

## What We're Building
在内容发送到 Memos 之前，增加一个“自动打标签”步骤。用户先在插件里配置一组候选标签，开启功能后，插件把待保存内容交给模型，在这组候选标签中挑选一个最合适的标签，再把该标签追加到 memo 内容中一起发送到 Memos。

这个功能必须是可选增强，而不是新的硬依赖。关闭时保持现有行为不变；开启但模型调用失败时也不能阻塞保存流程，而是直接跳过自动打标签，按原内容继续发送。

## Why This Approach
仓库里已经有现成的标签注入路径：`js/background.js` 会在快速保存时把 `quicksavetag` 追加到内容末尾，配置也已经通过 `chrome.storage.sync` 持久化。新需求的核心不是“如何写入标签”，而是“在发送前多一步可失败的标签选择”，所以最合理的方向是复用现有保存链路，在发送前插入一个轻量的预处理阶段。

考虑过三种方向：
- 只支持静态默认标签，用户自己维护 `quicksavetag`：改动最小，但不满足“模型在指定范围内选择”的新需求。
- 让模型直接生成任意标签：看起来灵活，但结果不可控，容易产生脏标签，也会偏离“只在指定范围内选一个”的边界。
- 用户提供候选标签列表，模型仅从中选择一个；调用失败或返回无效值就跳过：约束清晰、兼容现有实现，也是推荐方案。

推荐第三种。它把模型能力限制在“分类”而不是“生成”，既能提升命中率，也能把失败面压到最小。

## Key Decisions
- 自动打标签是显式开关：默认关闭，避免给现有用户引入隐性网络调用和新失败路径。
- 模型只允许从候选标签范围内选一个：不接受自由生成，确保标签集合可控。
- 模型配置独立于 Memos 配置：单独支持自定义 API Key、接口地址和模型名，不复用现有 Memos token。
- 调用失败默认跳过：包括请求失败、超时、返回空值、返回不在候选范围内，均不阻塞原始保存。
- 优先复用现有内容拼装方式：最终仍以“正文 + 来源链接 + 标签行”的形式发送，减少对现有 Memos 兼容性的影响。
- 配置形态保持轻量：沿用当前 popup 设置区的表单思路，新增开关、候选标签输入、模型三项基础配置，避免引入复杂设置页。
