# S3 Config Sync

## What We're Building
为浏览器扩展增加“配置自动同步到 S3”的能力，仅同步当前显式可导入导出的用户配置，也就是 [js/options.js](/Users/xiaohansong/projects/memos-bber/js/options.js) 中 `CONFIG_EXPORT_KEYS` 对应的字段。同步目标是让同一用户在多设备之间保持配置一致，同时不把运行时临时状态带入跨设备同步。

功能范围包括三部分：支持定时同步；同步前先比较内容，只有配置发生变化才写入远端；支持自动双向同步，即既能把本地变更推到 S3，也能在检测到远端更新时把配置拉回本地。除定时任务外，还需要在扩展启动时先执行一次拉取，在用户保存设置后立即执行一次推送，以缩短多端收敛时间。

## Why This Approach
仓库里已有明确的“配置边界”和“序列化格式”基础：设置页通过 `chrome.storage.sync` 保存配置，并且已有导出/导入 JSON 的能力。最合理的方向不是重新定义一套配置模型，而是基于现有导出白名单，把这份受控配置作为唯一同步对象，再在其外层增加远端版本协调逻辑。

考虑过三种方向：
- 仅做单向备份到 S3：实现最简单，但不满足自动双向同步。
- 同步 `chrome.storage.sync` 全量键值：维护成本低，但会把 `open_action`、`open_content` 一类运行时状态混入远端，带来脏数据和误覆盖风险。
- 仅同步 `CONFIG_EXPORT_KEYS` 白名单，并基于最后修改时间做双向同步：边界清晰、能复用现有导入导出语义，也能满足多端自动收敛需求。

推荐第三种。它在功能完整性和行为可控性之间最平衡，复杂度也仍然可控。

## Key Decisions
- 仅同步 `CONFIG_EXPORT_KEYS`：只同步显式用户配置，不同步运行时临时状态。
- 采用插件直连 S3：扩展内保存长期 S3 凭证，直接访问对象存储，不依赖中转服务。
- 支持定时同步：作为后台周期性检查机制，补足“启动拉取”和“保存推送”之间的时间窗口。
- 启动时自动拉取一次：浏览器启动或扩展 service worker 被唤醒时，优先尝试从远端拉取最新配置。
- 设置保存后立即推送一次：用户在设置页保存成功后，尽快把变更写到 S3。
- 仅在内容变化时同步：本地推送前先比较当前配置与最近一次已同步快照；拉取时先比较远端内容与本地配置，避免无意义覆盖。
- 双向同步按最后修改时间解决冲突：本地与远端都发生变化时，默认由较新的配置覆盖较旧配置。
- 远端对象需要携带元信息：至少包含配置内容、内容摘要和最后修改时间，供比较和冲突判定使用。
- 同步目标是“自动收敛”，不是“版本管理”：暂不提供历史版本浏览、回滚或复杂冲突 UI。
